# 09 — AI 系統（大名 AI 與委任 AI）

> 遵循 `plan/00-foundations.md`（下稱 00）之全部規範。實體與欄位命名以 00 §4/§8/§14 與
> `plan/02-data-model.md` 為準；數值常數一律以 `BAL.*` 命名，定案值以 `plan/15-balance.md` 為準。

---

## 1. 目的與範圍

本文件是下列機制的**單一真相來源**：

1. **敵方大名 AI**：所有非玩家勢力（`Clan`）的完整決策系統——每月「評定」戰略決策、
   每日反應式決策、外交決策、資源分配。
2. **委任 AI（玩家側）**：領主（`stewardId`）自動開發、委任城主（`lordId`）自動建造與徵兵、
   軍團（`Corps`）AI。
3. **具申生成**：每月為夠格家臣生成候選提案（`Proposal`）並評分送交玩家。
4. **難易度**對 AI 的修正方式。
5. AI 的效能預算、排程器、快取策略。
6. AI 決策的可測試性介面（`AiIntent`）。

**不在本文件範圍**：具申的受理／回饋／功績結算（見 `plan/06-officers.md`）、
外交接受度公式與調略成功率（見 `plan/08-diplomacy.md`）、
戰鬥解算與軍團實體定義（見 `plan/07-military.md`）、
開發／徵兵／施設的效果與成本（見 `plan/05-domestic.md`）。

## 3. 設計細節

### 3.1 總架構

AI 分為**四層**，全部實作於 `src/core/systems/ai/`：

| 層 | 頻率 | 內容 | 模組 |
|---|---|---|---|
| 戰略層（評定） | 每月一次（攤平執行，見 §3.3） | 威脅評估、擴張目標、防衛部署、資源分配、外交 | `ai/daimyoAi.ts` |
| 反應層 | 每日 | 領土被侵入、圍城解圍、合戰發動、援軍請求回應 | `ai/reactiveAi.ts` |
| 委任層 | 每月＋每日輕量 | 領主開發、委任城主建造徵兵、軍團攻防 | `ai/stewardAi.ts` / `ai/castleAi.ts` / `ai/corpsAi.ts` |
| 具申層 | 每月 1 日 | 為玩家家臣生成提案 | `ai/proposalGen.ts`（由 `systems/proposals.ts` 呼叫） |

**鐵律（canonical）**：

1. **AI 只能透過 Command 行動**。AI 產生的 `Command` 與玩家操作走完全相同的驗證器與 `apply`
   流程；驗證失敗的 Command 直接丟棄並記錄 `AiIntent`（`kind: 'error.invalidCommand'`）。
   AI **沒有任何**繞過規則的特權操作；難易度加成（§3.9）只作用於收入乘數、士氣加成、
   評定頻率與協同上限，不新增特權 Command。
2. **AI 可讀取完整 `GameState`**（v1.0 無戰爭迷霧系統；理由見 §8-D1）。
3. **決定論**：AI 內所有隨機皆取自 `rng.ai` 流（00 §5.5）；同分裁決（tie-break）
   一律先以分數降冪、再以 ID 字典序排序後才取亂數，確保重放一致。
4. AI 決策一律輸出 `AiIntent` 紀錄（§3.11），即使決策結果是「不行動」。

### 3.2 大名 AI 性格參數（`AiPersona`）

每個 AI 勢力有一組五軸性格參數（0..100，型別見 §4.1）：

| 軸 | 欄位 | 影響 |
|---|---|---|
| 侵攻性 | `aggression` | 開戰所需戰力比門檻（越高越敢打）、擴張分數乘數、攔截積極度 |
| 外交傾向 | `diplomacy` | 外交工作預算比例、主動求和／結盟意願 |
| 內政傾向 | `development` | 內政對軍事的資源分配比重、施設建造積極度 |
| 義理 | `loyalty` | 守約傾向：高者不毀約、積極回應援軍請求；低者傾向調略與背盟 |
| 野心 | `ambition` | 從屬意願（反向）、可同時進行的攻略計畫規模、對天下的目標感 |

**衍生傾向（不設獨立軸，公式見 §5.1）**：
調略傾向 `plotDrive =（aggression +（100 − loyalty））/ 2`——例如武田家高侵攻＋低義理
即自然成為「高調略」勢力。

### 3.3 評定排程器（amortization）

評定（戰略層決策）成本高（威脅評估需走訪節點圖），因此：

1. **攤平到全月**：每勢力有固定 `councilOffset = fnv1a(clanId) % BAL.aiCouncilSpreadTicks`
   （`BAL.aiCouncilSpreadTicks = 30`），其評定於每月第 `councilOffset + 1` 日開始，
   40 家自然散布在整個月，避免每月 1 日 tick 尖峰。
2. **單一勢力的評定再攤平為 4 個階段**（`BAL.aiCouncilPhaseCount = 4`），
   每 tick 只執行一個階段，連續 4 個 tick 完成：
   - Phase A `threat`：威脅評估（重算 `ThreatCache`）。
   - Phase B `military`：防衛部署 → 攻略計畫維護 → 新攻略計畫選定。
   - Phase C `domestic`：徵兵方針 → 施設建造 → 開發方針 → 褒賞 → 政策採用。
   - Phase D `diplomacy`：求和／同盟／壓迫／從屬／援軍回應／調略。
3. `fnv1a` 為標準 FNV-1a 32-bit 字串雜湊（實作於 `ai/hash.ts`），確保決定論且不需序列化排程表。
4. 上級難易度每月有兩次評定（§3.9）：第二次於第 `(councilOffset + 15) % 30 + 1` 日開始，
   只執行 Phase A＋B（臨時軍議）。

### 3.4 戰略層（每月評定）

#### 3.4.1 威脅評估（Phase A）

對每個**接壤**勢力（存在我方城／郡與其城／郡相鄰的 `RoadEdge`）計算威脅分（公式 §5.2），
結果存入 `ThreatCache`，有效期 `BAL.aiThreatCacheDays = 10` 日；期間內反應層直接讀快取，
但下列 `GameEvent` 立即使快取失效：我方或接壤勢力的城易主、對我宣戰、協定締結／破棄、
我城被圍。

#### 3.4.2 防衛（Phase B 前段）

對 `ThreatCache` 中 `threat ≥ BAL.aiThreatAlertRatio = 1.2` 的敵勢力，
其威脅所及的我方城（`threatenedCastleIds`）依序處理：

1. 該城守兵若低於預期敵方投入兵力 × `BAL.aiDefenseGarrisonRatio = 0.6`，
   從「安全城」（自身不在任何 `threatenedCastleIds` 內、且行軍 ≤ `BAL.aiReinforceMaxDays = 6` 日）
   下達輸送／出陣增援 Command，每座安全城最多抽調現有兵力 × `BAL.aiReinforceRatio = 0.5`。
2. 若總威脅（各接壤敵威脅分之和）≥ `BAL.aiRecallThreat = 2.0`，
   中止所有處於 `muster` / `advance` 階段的攻略計畫（`siege` 階段不中止，沉沒成本考量見 §8-D6），
   召回部隊回防。

#### 3.4.3 擴張目標選定（Phase B 後段）

對每個「可攻擊」的敵方城（與我接壤、非同盟／停戰對象；宣戰規則見 08）計算擴張分數
（公式 §5.3，即題旨 `score = 目標城價值 ×弱度 ÷ 距離`）。選定條件：

- 分數 ≥ `BAL.aiExpandScoreMin = 12`；
- 我方可投入兵力 ÷ 目標防衛兵力 ≥ 開戰門檻 `requiredRatio`（隨 `aggression` 下降，§5.3）；
- 進行中攻略計畫數 < 上限 `maxPlans = min(3, 1 + floor(castleCount / BAL.aiPlanPerCastles))`
  （`BAL.aiPlanPerCastles = 8`）。

選定後建立 `AttackPlan` 進入**攻略計畫狀態機**：

```
muster(集結) ──兵力到齊──▶ advance(進軍) ──抵達──▶ siege(圍城) ──落城──▶ consolidate(鞏固) ──▶ 計畫完成
   │timeout                    │戰力比惡化              │解圍軍不敵            │BAL.aiConsolidateDays
   ▼                           ▼                        ▼                     （期滿或無威脅提早結束）
 aborted ◀────────────────── aborted ◀────────────── aborted
```

- **muster（集結）**：從最多 `BAL.diffAiCoopMaxCastles`（依難易度 2/3/4）座我方城，
  以「距集結城行軍天數」升冪挑選出兵城（**多城協同出兵**），各城保留守備底線
  （`max(BAL.aiHomeGarrisonMin = 500, 現有兵力 × BAL.aiHomeGarrisonRatio = 0.3)`）後，
  出陣至集結城（`stagingCastleId`＝我方距目標最近的城）。
  超過 `BAL.aiMusterTimeoutDays = 30` 日未到齊 → 以已到兵力重新檢查開戰門檻，不足則 aborted。
- **advance（進軍）**：合流後對目標城下達出陣 Command（路徑由 04 尋路）。
  途中每日檢查：我方兵力 ÷（目標守兵＋偵測到的敵野戰部隊）< `BAL.aiAdvanceAbortRatio = 0.8`
  → aborted 撤退回集結城。
- **siege（圍城）**：抵達後依 07 進入攻城。模式選擇：
  我方兵力 ≥ 守兵 × `BAL.aiAssaultRatio = 3.0` 且城耐久 ≤ 最大值 × 0.5 → 強攻，否則包圍。
  敵解圍軍出現時交由反應層（§3.6）。攜行兵糧將於
  `BAL.aiSiegeFoodAbortDays = 15` 日內耗盡且無法補給 → aborted。
- **consolidate（鞏固）**：落城後留駐 `min(新城最大兵力, 攻方兵力 × BAL.aiConsolidateGarrisonRatio = 0.5)`，
  其餘部隊返回各出兵城；期間 `BAL.aiConsolidateDays = 45` 日該計畫佔用一個計畫槽
  （防止無限連續擴張），對新領地下達開發方針（新領治安低，依 §5.7 排除兵舍方針）。

#### 3.4.4 資源決策（Phase C）

依固定優先序執行（全部為 05／06／08 定義的 Command）：

1. **兵糧安全存量**：每座城兵糧目標 ≥ 守兵 × 每日糧耗 × `BAL.aiFoodReserveDays = 180`；
   不足者從有餘裕的城輸送（05 輸送 Command）。
2. **徵兵方針**：每城守兵目標 = 城最大兵力 ×（承平 `BAL.aiGarrisonTargetPeace = 0.7`；
   交戰中或受威脅 `BAL.aiGarrisonTargetWar = 0.95`）。低於目標且該城所轄郡治安
   ≥ `BAL.aiConscriptSecurityMin = 50` 時下達徵兵 Command（數量與人口約束見 05）。
3. **褒賞維穩**：忠誠 < `BAL.aiRewardLoyaltyThreshold = 40` 的家臣，
   依（身分高者優先、同身分忠誠低者優先）排序，逐一下達褒賞 Command，
   當月褒賞總支出 ≤ 可用金錢（扣除儲備，見下）× `BAL.aiRewardBudgetRatio = 0.3`。
4. **施設建造**：金錢儲備 = 月收入 × `BAL.aiGoldReserveMonths = 2`；超出儲備的部分
   × `BAL.aiBuildBudgetRatio = 0.5` 為建造預算。優先序以施設 `effectTag`（05 定義）決定：
   受威脅城 → `defense` > `military` > `food`；安全城 → `economy` > `food` > `military` > `loyalty`。
   同 tag 取 05 表列順序第一個可建者。
5. **政策採用**：威信與金錢滿足條件的政策中，選「persona 加權效益」最高者
   （`economy` 類 × development/50、`military` 類 × aggression/50、`diplomacy` 類 × diplomacy/50）。
6. **開發分配**：AI 勢力的郡與玩家委任的郡走同一套領主自動開發（§3.7.1）；
   直轄郡由本階段以相同演算法下達開發方針。

#### 3.4.5 外交層（Phase D）

依序評估（各行動的成本、信用消耗、**接受度公式**皆引用 08，本節只定義 AI 何時發起／接受）：

1. **對強鄰求和／同盟**：對 `threat ≥ BAL.aiDiploFearThreshold = 1.5` 的交戰或無協定鄰國，
   若 `diplomacy ≥ 40`：先下達外交工作 Command 累積信用；信用足夠時，交戰中→請求停戰，
   非交戰→請求同盟。發起意願分 `= threat × diplomacy / 100`，取最高者一國／月。
2. **對弱鄰壓迫（從屬勸告）**：對戰力比（我÷彼）≥ `BAL.aiVassalDemandRatio = 3.0` 的鄰國
   發出從屬勸告 Command（接受與否由對方依 08 公式判定）。
3. **受理從屬／同盟／停戰**（AI 為受方時，於收到請求的當日反應層立即判定）：
   一律呼叫 08 的接受度公式；本文件補充 AI 專屬修正項——
   從屬受理意願加成 `=（100 − ambition）× BAL.aiVassalAmbitionWeight = 0.4`（加到 08 公式的意願項）。
4. **援軍請求響應**：收到同盟方援軍請求時計分
   `aidScore =（loyalty / 100）× allianceFactor × safetyFactor`（§5.6），
   ≥ `BAL.aiAidScoreMin = 0.5` → 依 §3.4.3 的協同出兵邏輯派兵（目標＝請求方指定城）。
5. **背盟與毀約**：`loyalty ≥ BAL.aiBetrayLoyaltyMax = 40` 的勢力**永不**主動破棄協定。
   低於此值者：停戰破棄需戰力比 ≥ 2.0；同盟破棄需 `loyalty < 25` 且戰力比 ≥ 3.0
   （破棄的信用懲罰見 08）。
6. **調略**：`plotDrive ≥ 60` 的勢力每月挑一個目標下達調略 Command——
   優先序：敵方忠誠最低且 < 50 的城主 → 引拔；圍攻目標城 → 內應；威脅最大敵國 → 流言
   （成功率與效果見 08）。

### 3.5 反應層（每日）

反應層讀取**上一 tick 產生的 `GameEvent`**（03 事件匯流排）與當前狀態，成本必須輕量
（只允許讀 `ThreatCache` 與局部鄰域，禁止全圖走訪）：

1. **領土被侵入**：敵 `Army` 進入我方郡／城節點 → 找出距其最近的我方有兵之城，
   估算戰力比（§5.5）；我 ÷ 敵 ≥ `BAL.aiInterceptRatioMin = 1.1` → 出兵攔截（出陣 Command，
   目標＝敵軍當前節點）；不利則固守（不出兵，僅記錄 `AiIntent`，`kind: 'defense.hold'`）。
2. **我城被圍 → 解圍軍**：從行軍 ≤ `BAL.aiSiegeReliefMaxDays = 8` 日的我方城集結解圍兵力，
   合計 ≥ 圍城敵軍 × `BAL.aiSiegeReliefRatio = 1.2` 才出動（多城各自出陣、目標同一節點）；
   湊不足則不出動（守城消耗戰）。
3. **合戰發動決策**：我方 `Army` 與敵接戰且滿足 07 的合戰發動條件（兵力門檻等）時，
   計算預測勝率（§5.5）> `BAL.aiKassenWinRateMin = 0.65` → 下達合戰發動 Command
   （AI vs AI 的合戰一律自動解算；AI 對玩家發動時依 07 呈現）。反之維持野戰自動解算。
4. **外交來使回覆**：見 §3.4.5-3（受方判定即時執行，不等評定）。

### 3.6 圍城解圍的攻方應對

我方攻城中、敵解圍軍接近至 ≤ 3 節點時（每日檢查）：
圍城軍 ÷（解圍軍＋城內守兵 × 0.5）≥ 1.0 → 分兵迎擊（維持圍城最低兵力 = 守兵 × 1.5）；
否則解圍軍抵達前撤圍退回集結城（計畫 → aborted）。

### 3.7 委任 AI（玩家側；AI 勢力共用同一套實作）

委任 AI 花費的是玩家勢力資源，因此有**護欄**：不得使金錢低於儲備
（月收入 × `BAL.aiGoldReserveMonths`）、不得動用大命、不得發起外交協定與調略
（具申才能建議這些，決定權在玩家）。

#### 3.7.1 領主自動開發（`stewardAi.ts`）

受封郡的領主每月 1 日選定該郡當月開發方針（05 的開發方針 Command）。
選法＝「短板與潛力」評分（§5.7）：開發重點三選一＝`agri`／`commerce`／`barracks`
（依 02 `DevelopFocus`，第三值為兵舍，E-07）；治安 < `BAL.aiStewardSecurityFloor = 60` 時
排除 `barracks`（避免對治安不穩郡加大人口／徵兵壓力），於石高／商業取
`gap =（潛力上限 − 現值）/ 潛力上限` 最大者；否則於石高／商業／人口三屬性取 gap 最大者。AI 勢力的直轄郡與玩家直轄但設為
「自動」的郡走同一函式。每日的開發步進本身由 `systems/development.ts` 執行（00 §5.4 第 5 步）。

#### 3.7.2 委任城主自動建造與徵兵（`castleAi.ts`）

玩家設定 `directControl = false` 的城，其城主每月執行：

1. **建造**：依 §3.4.4-4 的同一 `effectTag` 優先序表（受威脅判定改用玩家勢力的 `ThreatCache`；
   玩家勢力同樣每月計算 `ThreatCache` 供委任 AI 與具申生成使用，但不做評定決策）。
2. **徵兵**：同 §3.4.4-2 的守兵目標邏輯。
3. **兵糧**：低於安全存量時向玩家勢力最近的有餘城發輸送 Command。

#### 3.7.3 軍團 AI（`corpsAi.ts`）

軍團（`Corps`，實體與方針欄位見 07）由軍團長依方針行動，**共用大名 AI 的攻略計畫狀態機**
（同一 `AttackPlan` 型別與轉移規則），差異：

- 範圍限定：目標城、出兵城皆必須屬於軍團領（`Corps` 轄下城集合）或與軍團領接壤的敵城。
- 方針對應（02 §3.3 `CorpsDirective`＝`advance`／`hold`／`develop`；目標節點 `targetNodeId`，E-21）：
  - **`advance`（攻略）**：執行 §3.4.3 擴張選定（`aggression` 以玩家勢力 persona 視為 70 計）＋防衛；
    `targetNodeId` 非空時以其為首要攻略目標。
  - **`hold`（固守）**：只執行 §3.4.2 防衛與 §3.5 反應層，不建新攻略計畫。
  - **`develop`（開發）**：不建攻略計畫、不主動出擊，僅執行軍團領內政（§3.7.1／§3.7.2）與 §3.5 反應層防衛。
- 軍團的內政（軍團領內委任城）自動套用 §3.7.1／§3.7.2。
- 軍團評定排程：於每月第 `fnv1a(corpsId) % 30 + 1` 日執行，同樣分 4 階段攤平。

### 3.8 具申生成（`proposalGen.ts`，每月 1 日）

於 tick 第 10 步 `proposals`（00 §5.4）呼叫。對玩家勢力（AI 勢力不生成具申）：

1. **資格**：身分 ≥ 侍大將（`Rank` 見 06）、未捕虜、未出陣中、
   距其上次具申被採納或駁回 ≥ `BAL.proposalCooldownDays = 60` 日。
2. **候選生成**（每人最多 1 件）：取該武將四維中最高且 ≥ `BAL.proposalStatThreshold = 70` 的能力：
   - `pol` 最高 → **開發／建設案**：目標＝其知行郡（無知行則所屬城轄郡）中 §5.7 gap 最大者
     → 生成開發方針 Command；或所屬城依 §3.4.4-4 優先序可建施設 → 建造 Command。
   - `int` 最高 → **調略／外交案**：玩家勢力 `ThreatCache` 最大威脅國 → 依 §3.4.5-6 優先序
     生成調略 Command；或對威脅 ≥ 1.5 之國生成外交工作 Command。
   - `ldr` 或 `val` 最高 → **出陣案**：以 §5.3 對玩家勢力算擴張分數，取最高目標
     生成出陣 Command（出兵城＝該武將所在城）。
3. **評分**：`proposalScore = 能力值 × 局勢需求係數 needFactor`（§5.8），
   全勢力排序取前 `BAL.proposalTopK = 3` 件送給玩家（送達、受理、功績結算見 06）。
4. **合法性保證**：提案 payload 必須是**當下即通過 Command 驗證器**的合法 Command；
   生成時實際呼叫驗證器過濾，不合法者棄置。玩家批准時若情勢已變導致驗證失敗，
   依 06 的失效流程處理。

### 3.9 難易度

難易度只修正 AI 勢力、永不改變玩家規則（00 §11）。定案表在 15，建議值：

| 常數 | 初級 easy | 中級 normal | 上級 hard | 作用點 |
|---|---|---|---|---|
| `BAL.diffAiIncomeMult` | 0.8 | 1.0 | 1.2 | AI 勢力金錢與兵糧收入乘數（economy 系統套用） |
| `BAL.diffAiMoraleBonus` | 0 | 0 | +10 | AI 部隊與城的士氣加成（07 套用，上限仍 100） |
| `BAL.diffAiCouncilTimesPerMonth` | 1 | 1 | 2 | 評定頻率（§3.3-4 臨時軍議） |
| `BAL.diffAiCoopMaxCastles` | 2 | 3 | 4 | 攻略計畫協同出兵城數上限（§3.4.3） |

實作介面：`getDifficultyMods(state): { incomeMult, moraleBonus, councilTimes, coopMax }`，
存檔內記錄難易度設定（16）。

### 3.10 效能

1. **tick 預算**：AI 步驟（tick 第 11 步）軟預算 `BAL.aiTickBudgetMs = 4` 毫秒。
   排程器持有一個「本 tick 待辦」佇列（當日應執行評定階段的勢力＋反應層檢查）；
   以 `performance.now()`（core 內以注入的 `now()` 取得，測試環境用假時鐘）量測，
   超過預算時將剩餘**評定階段**順延到下一 tick（反應層不順延，永遠當日完成）。
   順延不影響決定論：順延判斷只依據佇列長度上限 `BAL.aiCouncilMaxPerTick = 4`
   （每 tick 最多執行 4 個評定階段），**不**依據實際毫秒——毫秒量測僅供 debug 面板顯示
   （理由見 §8-D7）。
2. **40 家攤平**：§3.3 的 `councilOffset` 把 40 家散布於 30 tick；期望每 tick
   1~2 家 × 1 階段，配合 `BAL.aiCouncilMaxPerTick` 削峰。
3. **快取**：`ThreatCache`（§3.4.1）＋尋路距離快取（`estimateMarchDays` 結果以
   `(fromCastleId, toCastleId)` 為 key 快取，城易主或道路狀態變更時失效；快取實作於 04）。
4. **反應層輕量化**：只在存在相關 `GameEvent`（侵入／圍城／接戰／來使）時才執行對應檢查；
   無事件的 tick，反應層 O(1) 直接返回。

### 3.11 可測試性（`AiIntent`）

所有 AI 決策點輸出 `AiIntent`（型別 §4.5）到 `state.ai.intentLog`（環形緩衝，
容量 `BAL.aiIntentLogSize = 1024`）。用途：

- **Debug 面板**（01 的除錯工具）：依勢力／層級篩選顯示最近決策與分數分解。
- **單元測試**（17）：建構特定局面 → 跑一次評定 → 斷言 `intentLog` 含預期 `kind` 與
  `commands`。例：「接壤弱國、戰力比 2.0、aggression 90 → 必產生 `expand.select`」。
- `intentLog` **不序列化進存檔**（decisions 已反映在 Command 紀錄與狀態中；§8-D5）。

## 5. 演算法與公式

本節所有除法以 `max(分母, 1)` 防除零；所有比較為含等號時明寫 `≥`。
本節公式中的 `persona`／`ai.persona` 均指解析後性格＝`state.ai.personas[ai.personaId]`（E-60）。

### 5.1 persona 衍生量

```
plotDrive     = (persona.aggression + (100 - persona.loyalty)) / 2      // 調略傾向 0..100
expandDrive   = 0.5 + persona.aggression / 100                          // 擴張分數乘數 0.5..1.5
requiredRatio = BAL.aiAttackPowerRatioBase - persona.aggression / 100 * BAL.aiAttackPowerRatioAggrBonus
                // 開戰所需戰力比；BAL.aiAttackPowerRatioBase = 1.8,
                // BAL.aiAttackPowerRatioAggrBonus = 0.6 → aggression 90 ⇒ 1.26
devBudgetShare = 0.35 + persona.development / 100 * 0.3                 // 內政預算佔比 0.35..0.65
```

### 5.2 威脅評估（Phase A）

```
computeThreatCache(state, clanId):
  entries = []
  for enemy in 與 clanId 接壤的勢力（非自身；含從屬主家視為接壤）:
    borderCastlesE = enemy 方與我領接壤或距邊界 ≤ 2 節點的城
    borderCastlesO = 我方與 enemy 領接壤或距邊界 ≤ 2 節點的城
    borderPowerEnemy = Σ estPower(城)  for 城 in borderCastlesE
                     + Σ estPower(部隊) for enemy 部隊位於我領或邊界 2 節點內
    borderPowerOurs  = Σ estPower(城)  for 城 in borderCastlesO
    relationFactor = 交戰中 ? 1.5 : 無協定 ? 1.0 : 停戰 ? 0.6 : (同盟|從屬) ? 0.2
    threat = borderPowerEnemy / max(borderPowerOurs, 1) * relationFactor
    threatenedCastleIds = borderCastlesO 中距 enemy 領 ≤ 1 節點者
    entries.push({...})
  排序 entries 依 threat 降冪、enemyClanId 字典序
  return { computedDay: today, entries, totalThreat: Σ threat }
```

`estPower`（AI 估算用近似值，**非** 07 的戰鬥公式；刻意粗糙以模擬判斷誤差）：

```
estPower(城)   = 城兵力 × (1 + 城主.ldr / BAL.aiPowerLdrDivisor) × (城士氣 / 100)
estPower(部隊) = 部隊兵力 × (1 + 主將.ldr / BAL.aiPowerLdrDivisor) × (部隊士氣 / 100)
// BAL.aiPowerLdrDivisor = 240 → ldr 120 的名將使估值 ×1.5
```

### 5.3 擴張目標分數（Phase B）

題旨公式 `score = 目標城價值 × 弱度 ÷ 距離` 具體化：

```
value(城) = Σ(城轄各郡 kokudaka) / BAL.aiValueKokudakaDivisor        // BAL.aiValueKokudakaDivisor = 1000
          + degree(城節點) * BAL.aiValueConnectivityWeight            // 位置價值：街道連接數，權重 2
          + (城.tier == 'main' ? BAL.aiValueMainCastleBonus : 0)      // 本城加成 15

weakness = clamp( ourCommittablePower / max(estPower(目標城) + 鄰援(目標城), BAL.aiWeaknessEpsilon),
                  0, BAL.aiWeaknessCap )
           // ourCommittablePower：協同上限內各城扣守備底線後可出兵力的 estPower 總和
           // 鄰援：目標城 1 節點內同勢力城可馳援兵力 × 0.5
           // BAL.aiWeaknessEpsilon = 100, BAL.aiWeaknessCap = 3.0

distanceDays = estimateMarchDays(stagingCastle, 目標城)               // 04 尋路
distFactor   = max(1, distanceDays / BAL.aiDistanceNormDays)          // BAL.aiDistanceNormDays = 10

score = value * weakness / distFactor * expandDrive
```

選定流程（Phase B 後段）：

```
selectExpansion(state, ai):
  if ai.attackPlans.length ≥ maxPlans: return intent('expand.skip', {reason:'planFull'})
  candidates = 可攻擊敵城（接壤、非同盟/停戰；含交戰中勢力全部接壤城）
  for c in candidates: 計算 score(c)
  排序：score 降冪 → castleId 字典序
  best = 首個滿足 score ≥ BAL.aiExpandScoreMin 且
         ourCommittablePower / (estPower(best)+鄰援) ≥ requiredRatio 者
  if 無: return intent('expand.none')
  建立 AttackPlan(stage='muster')，對 sourceCastleIds 下達出陣（至集結城）Command
  return intent('expand.select', {target: best.id}, scores=[value, weakness, distFactor, score])
```

### 5.4 攻略計畫狀態機（每日維護，`ai/attackPlan.ts`）

```
tickPlan(state, ai, plan):                       // 大名與軍團共用；每日對每個 plan 執行
  if 目標城已屬我方 → stage = 'consolidate'（若尚未），依 §3.4.3 留駐與回程
  if 目標城屬第三方（被他勢力先佔）→ abort('targetLost')
  if plan.sourceCastleIds 有城易主 → abort('sourceLost')
  switch plan.stage:
    'muster':
      if 各路部隊皆抵達集結城:
        if 合計兵力 ≥ plannedTroops * 0.8 → 下達對目標出陣 Command；stage='advance'
        else if 合計 estPower / 目標防衛 ≥ requiredRatio → 同上（將就出發）
        else → abort('musterWeak')
      else if today - stageEnteredDay > BAL.aiMusterTimeoutDays → 同上重檢，不足則 abort('musterTimeout')
    'advance':
      if 我 estPower / (目標守 estPower + 沿途敵野戰 estPower) < BAL.aiAdvanceAbortRatio
        → 下達回軍 Command；abort('advanceOutmatched')
      if 部隊抵達目標節點 → stage='siege'（攻城由 07 系統接手；模式選擇見 §3.4.3）
    'siege':
      §3.6 解圍應對；兵糧檢查（BAL.aiSiegeFoodAbortDays）
      落城 → stage='consolidate'
    'consolidate':
      if today - stageEnteredDay ≥ BAL.aiConsolidateDays or 新城無威脅(threat<0.8)
        → 計畫完成，自 attackPlans 移除
  abort(reason): 下達回軍 Command（各部隊返最近我方城）；記 intent('plan.abort',{reason})；移除計畫
```

### 5.5 反應層計算

```
// 攔截判定（收到「敵軍入侵我領」事件）
interceptCheck(state, ai, enemyArmy):
  myCastle = 距 enemyArmy 節點最近、守兵 > BAL.aiHomeGarrisonMin 的我方城
  ratio = estPower(myCastle 可出兵力) / estPower(enemyArmy)
  if ratio ≥ BAL.aiInterceptRatioMin → 出陣攔截；intent('defense.intercept')
  else → intent('defense.hold', {ratio})

// 合戰預測勝率（07 發動條件滿足時）
predictedWinRate = ourEst / (ourEst + enemyEst)
  其中 ourEst = estPower(我參戰各隊總和) × (1 + 地利加成 0.1 若我為守方)
  enemyEst 同理（AI 勢力士氣含 BAL.diffAiMoraleBonus）
發動條件：predictedWinRate > BAL.aiKassenWinRateMin（= 0.65）
```

### 5.6 援軍請求響應分

```
aidScore = (persona.loyalty / 100) × allianceFactor × safetyFactor
  allianceFactor：同盟 1.0；婚姻同盟 1.2；僅停戰/從屬關係 0.4
  safetyFactor = clamp(1.5 - threatCache.totalThreat / 2, 0, 1)   // 自身越危險越不出兵
派兵條件：aidScore ≥ BAL.aiAidScoreMin（= 0.5）
派兵規模：可出兵力 × min(aidScore, 1) × 0.6
```

### 5.7 領主開發選方針（短板×潛力）

pickDevFocus(district):                              // 回傳 DevelopFocus∈{agri, commerce, barracks}（依 02，E-07）

## 8. 設計決策記錄

- **D1　AI 讀取完整狀態、不作弊操作**：v1.0 無戰爭迷霧系統，AI 直接讀真實 `GameState`；
  「公平性」由「AI 只能下合法 Command」保證。`estPower` 刻意用粗糙近似式而非 07 精確戰鬥公式，
  讓 AI 判斷帶有擬真誤差，也避免複述 07 公式（00 §7 單一定義原則）。
- **D2　persona 五軸不含「調略」軸**：任務規格定義五軸；調略傾向以
  `plotDrive = f(aggression, loyalty)` 衍生，能自然表現「武田高調略、長尾不屑調略」，
  少一個需要 40 家逐一調的參數。
- **D3　攻略計畫狀態機由大名 AI 與軍團 AI 共用**：同一 `AttackPlan` 型別加 `ownerCorpsId`
  區分歸屬與範圍限制，避免兩套平行實作漂移。
- **D4　評定攤平採「councilOffset＋4 階段」雙層攤平**：40 家 × 4 階段 = 160 階段散布於 30 tick，
  均值 5.3 階段/tick，配合 `aiCouncilMaxPerTick = 4` 與 deferred 佇列削峰；
  各勢力評定日期彼此錯開也讓世界節奏更自然（不會全天下同日總動員）。
- **D5　`AiIntent` 不序列化**：決策結果已完整反映於 Command 紀錄與狀態，重放即可重建；
  序列化只會膨脹存檔。golden test 斷言改在重放過程中收集 intentLog。
- **D6　`siege` 階段不因後方威脅而中止**：圍城已付出行軍與糧耗沉沒成本，且撤圍常導致
  半途而廢的振盪行為（打→撤→再打）；僅 `muster`／`advance` 受 `aiRecallThreat` 影響。
- **D7　效能預算不以實際毫秒做行為分歧**：以毫秒判斷會使不同機器產生不同決策序列，
  破壞決定論；改用固定的 `aiCouncilMaxPerTick` 計數削峰，毫秒量測僅供 debug 顯示。
- **D8　施設建造優先序以 `effectTag` 分類而非列舉施設名**：施設清單屬 05 的真相範圍，
  本文件以 tag 排序表達策略，05 為每個施設標注 `effectTag` 即可，兩文件不互相複述清單。
- **D9　40 家 persona 表為建議值、勢力清單以 14 為準**：資料製作（14）可能微調勢力數（38–42），
  以「查表＋預設值 fallback」設計使兩文件解耦；數值定案仍歸 15 主表。
