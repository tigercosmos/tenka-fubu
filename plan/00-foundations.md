## 0. 如何使用本計畫（給實作 AI 的指示）

本計畫由多份規格文件構成，目標是讓實作者（預期為 Claude Opus 4.8）**不需要再做任何遊戲設計決策**，
直接依規格寫程式即可。使用方式：

1. 先完整閱讀本文件與 `plan/README.md`。
2. 依 `plan/18-roadmap.md` 的里程碑順序實作。每個里程碑開始前，閱讀該里程碑指定的規格文件。
3. 所有數值常數以 `BAL.*` 常數名引用，實際值以 `plan/15-balance.md` 主表為準，實作於 `src/core/balance.ts` 單一檔案。
4. 所有 UI 文字必須是**繁體中文（台灣慣用語）**，集中於 `src/i18n/zh-TW.ts`，key 規範見 §9。
5. 若實作時發現規格有矛盾或不可行之處：以優先序高的文件為準做出決定、完成實作，
   並回頭修改對應的 plan 文件、在該文件的「設計決策記錄」章節註記變更原因。**不得留下未解決的 TBD。**
6. 遊戲術語（統率、石高、威風……）的中文用字以 `plan/19-glossary.md` 為準，禁止出現簡體字。

## 1. 遊戲願景與範圍

### 1.1 一句話定位

一款在瀏覽器中執行的單機戰國大戰略遊戲，玩法向《信長之野望·新生》致敬：
玩家扮演戰國大名，透過**知行（把土地分封給家臣，家臣自主治理）**、內政開發、外交調略、
即時行軍與合戰，最終統一天下。

### 1.2 工作名稱

工作標題：**《天下布武》**（程式碼專案名 `tenka-fubu`）。
本作是**非商業的致敬同人作品**，不使用 KOEI 的美術、音樂、程式資產；
歷史人物與史實為公有領域，機制為重新設計實作。README 需載明此聲明。

### 1.3 核心體驗支柱（依重要性排序）

1. **知行與家臣自主**：玩家不微管理每一塊地；把郡分封給武將，武將自動開發，玩家做大方向決策。
2. **即時天下大勢**：全日本地圖上，各勢力同時即時行動（可暫停、變速），戰線與勢力消長一目瞭然。
3. **威風的爽快感**：打贏一場關鍵合戰，鄰近敵郡望風而降，戰線大幅推進。
4. **武將是資產**：登用、育成、任命、忠誠管理是長期經營核心。
5. **歷史沉浸**：史實事件、官位朝廷幕府、繁體中文的戰國語感。

### 1.4 明確納入範圍（v1.0）

- 單一劇本：**「1560 桶狹間前夜」**，全日本地圖（本州＋四國＋九州，不含蝦夷與離島）。
- 系統：時間流動、內政（郡開發／知行／城下施設／政策／徵兵）、軍事（出陣／行軍／野戰／合戰／攻城）、
  威風、外交（同盟／婚姻／停戰／從屬）、朝廷與幕府、調略（引拔／流言／內應）、
  具申、身分與功績、軍團委任、歷史事件、大命、勝敗判定、存讀檔。
- 敵方大名 AI、家臣（領主／城主／軍團）委任 AI。
- 純前端 SPA，無後端，可部署至 GitHub Pages。

### 1.5 明確排除範圍（v1.0 不做）

- 多人連線、雲端存檔、帳號系統。
- 3D 地形；本作採風格化 2D 向量地圖。
- 音樂音效（架構預留開關，v1.0 靜音）。
- 武將列傳百科、編輯器、成就系統、多劇本（架構需支援多劇本，但只做一個）。
- 行動裝置佈局（桌面優先，最小視窗寬 1280px）。

## 3. 程式架構與目錄結構（canonical）

最重要的架構鐵律：**`src/core/` 是純 TypeScript 遊戲邏輯，不得 import 任何 React／Pixi／DOM API。**
core 是一個確定性的狀態機：`(GameState, Command[]) --tick--> GameState + GameEvent[]`。
UI 層只做兩件事：把玩家操作轉成 `Command` 丟進佇列；訂閱狀態渲染畫面。

├── src/
│   ├── core/                   # 純邏輯，零 DOM 依賴（可在 Node 跑測試）
│   │   ├── state/              # GameState 型別、初始化 builder、selector
│   │   ├── commands/           # Command 型別聯集、驗證器、apply
│   │   ├── systems/            # tick 系統（依 §5.4 固定順序執行）
│   │   │   ├── time.ts         # 曆法推進

│   │   │   ├── development.ts  # 郡開發、領主自動治理

## 4. 核心概念模型（entity 一覽）

實體名稱與關係如下（欄位細節由 `02-data-model.md` 定義；**實體與欄位命名以 02 為準**，
下表的名稱為 canonical，不得另創同義詞）：

```
Clan(勢力/大名家) 1──* Officer(武將)
Clan 1──* Castle(城) 1──* District(郡)
Province(國，僅顯示分組) 1──* Castle
Castle ──街道邊(RoadEdge)── Castle/District   （地圖是節點圖：節點=城∪郡）
Clan 1──* Army(出陣中部隊)   Army ──> path: MapNodeId[]
Clan 1──1 DiplomacyRow(對每個其他 Clan：信用/感情/協定)
GameState = { time, clans, officers, castles, districts, provinces,
              armies, battles, diplomacy, court(朝廷/幕府), events, reports, rng }
```

關鍵設計事實（canonical）：

- **金錢是勢力層級**資源；**兵糧、兵力儲存在城**（攻城戰才有糧盡的張力）。
- **郡是最小土地單位**：擁有 石高、商業、人口、治安、開發度；可分封給武將成為**領主**，或維持**直轄**。
- **城分本城／支城**（城格）。城的最大兵力由所轄各郡人口推導。
- **武將四維能力**：統率、武勇、知略、政務（1..120，史實猛將可破百）。
- **身分六階**：足輕組頭 → 足輕大將 → 侍大將 → 部將 → 家老 → 宿老；功績累積升格；
  身分決定可任官職（侍大將以上可任城主；家老以上可任軍團長）與知行上限。
- **地圖即節點圖**：行軍沿街道邊移動；經過敵郡需花時間「制壓」翻轉歸屬。
- **合戰（戰術戰場）**是選擇性的：野戰預設自動解算，兵力達門檻時玩家可發動合戰進入戰術畫面；
  合戰大勝產生**威風**，使鄰近敵郡歸順、敵城士氣下降。

## 5. 時間與遊戲主迴圈（canonical）

### 5.1 曆法

- `1 tick = 1 日`；`1 月 = 30 日`；`1 年 = 12 月 = 360 日`。西曆紀年（如「1560年5月」）。
- 季節：春 3–5 月、夏 6–8 月、秋 9–11 月、冬 12–2 月。**9 月 1 日執行秋收**。

### 5.2 速度與暫停

- 速度檔位：`暫停 / ×1（600ms/日）/ ×2（300ms/日）/ ×5（120ms/日）`。
- 任何指令（Command）都可在任意時刻下達，於**下一個 tick 開頭**統一結算。
- 自動暫停事件（可於設定關閉）：我方城被圍、合戰可發動、具申送達、外交來使、歷史事件、月初。

### 5.3 主迴圈驅動（UI 層）

`requestAnimationFrame` 累加器：累積實際毫秒 ≥ 當前檔位的每日毫秒數時，呼叫 `core.advanceDay()` 一次。
合戰與歷史事件以 modal 呈現時，策略層時間暫停。

### 5.4 每日 tick 的系統執行順序（固定，確保決定論）

```
advanceDay(state, commandQueue):
  1. applyCommands        — 依提交順序套用本 tick 前收到的全部 Command
  2. time                 — 日期+1；觸發月初/月末/季/年 hooks
  3. events               — 歷史事件與汎用事件判定（月初為主）
  4. diplomacy            — 外交工作進度、協定到期
  5. development          — 郡開發進度（每日小步進）
  6. economy              — 兵糧消耗（每日）；收入（每月 1 日）；秋收（9/1）
  7. military.movement    — 部隊行軍、制壓、遭遇判定
  8. military.combat      — 野戰自動解算 tick、攻城 tick（合戰在 modal 內以子迴圈跑）
  9. officers             — 忠誠變化（每月）、壽命死亡（每月）、元服（每年 1/1）
 10. proposals            — 具申生成（每月 1 日）
 11. ai                   — AI 勢力決策（評定：每月 1 日；反應式：每日）
 12. victory              — 勝敗條件檢查
 13. reports              — 彙整本 tick 的 GameEvent → 通知
```

### 5.5 決定論

- 全部隨機數經 `rng.ts`（mulberry32），依用途分流：`rng.battle / rng.dev / rng.ai / rng.event / rng.misc`。
- 禁止在 core 內使用 `Date.now / Math.random`。
- 同一（劇本、種子、Command 紀錄）必須重放出完全相同的狀態 → golden test（見 17）。

**平衡設計目標（供 15-balance 推導常數，非直接實作值）：**

- 1 萬石 ≈ 可養 250 常備兵（`BAL.soldiersPerKoku ≈ 0.025`）。
- 10 萬石大名應能維持約 3,000 野戰兵力連續作戰 6 個月不斷糧。
- ×1 速度下，統一天下的一盤約 4–8 小時真實時間（約遊戲內 15–25 年）。
- 開局（1560）織田家：尾張半國 ~30 萬石規模；今川家 ~70 萬石（駿遠三）。

## 8. 資料與 ID 命名規範（canonical）

- **ID 一律小寫羅馬字 slug**，以型別前綴確保全域唯一：
  - 勢力 `clan.oda`、武將 `off.oda-nobunaga`、城 `castle.kiyosu`、郡 `dist.owari-kasugai`、
    國 `prov.owari`、街道邊 `road.kiyosu-sunpu-01`、事件 `evt.okehazama`、政策 `pol.rakuichi`、
    特性 `trait.gunshin`、戰法 `tac.charge`、大命 `taimei.sokuji`。
  - 同名武將以生年後綴消歧：`off.honda-tadakatsu-1548`（僅衝突時）。
- 顯示名稱（繁中）放在資料 JSON 的 `name` 欄位（如 `"織田信長"`、`"清洲城"`、`"春日井郡"`），
  **專有名詞不進 i18n 字串表**。
- 座標：地圖世界空間 4096×4096。由近似經緯度線性投影：
  `x = (lon − 128.5) / (146.0 − 128.5) × 4096`、`y = (45.8 − lat) / (45.8 − 30.5) × 4096`。
- 劇本資料檔一律 JSON、UTF-8、由 `src/data/schemas/` 的 zod schema 驗證（`tools/validate.ts`）。

## 9. i18n 與字串規範（canonical）

- 唯一語系 `zh-TW`，但所有 UI 文字必須經 `t(key)` 取得，不得硬編碼在元件中。
- Key 命名：`區域.畫面或元件.語意`，例：`ui.hud.gold`、`ui.castle.garrison`、
  `cmd.march.confirm`、`report.siege.fallen`、`term.kokudaka`。
- 插值用 `{name}` 大括號具名參數：`"report.battle.won": "{attacker}於{place}擊破{defender}！"`。
- 日期格式：`1560年5月3日`；數量單位緊貼數字：`3,000兵`、`12,500石`、`800貫`。
- **全案禁止簡體字**（驗證腳本會以常見簡體字元黑名單掃描，見 17）。

## 11. BAL 常數規範

- 所有可調數值（係數、成本、門檻、機率、時長）**必須**命名為 `BAL.camelCaseName` 並集中在
  `src/core/balance.ts`。系統規格文件中寫公式時引用常數名並附「建議初值」；
  `15-balance.md` 彙整成主表，若各文件建議值衝突，以 15 定案值為準。
- 難易度（初級/中級/上級）修正只能作用於 AI 勢力的收入/士氣/AI 積極度乘數，不改玩家規則。
