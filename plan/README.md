## 1. 目的與範圍

### 1.1 專案一句話

《天下布武》（程式碼專案名 `tenka-fubu`）是一款在瀏覽器中執行的**單機戰國大戰略遊戲**，
玩法向《信長之野望·新生》致敬：玩家扮演戰國大名，透過知行（把郡分封給家臣、家臣自主治理）、
內政開發、外交調略、即時行軍與合戰，最終統一天下（00 §1.1）。
純前端 SPA、無後端，部署至 GitHub Pages（00 §1.4）。

### 1.2 非商業致敬聲明

本作是**非商業的致敬同人作品**：不使用 KOEI 的美術、音樂、程式資產；
歷史人物與史實為公有領域；全部機制為重新設計實作（00 §1.2）。
專案根目錄的 README 與遊戲標題畫面皆須載明此聲明。

### 1.3 本文件的定位與讀者

- **讀者**：實作 AI（預期為 Claude Opus 4.8）與任何後續維護者。
- **定位**：`plan/` 目錄共 **21 份文件**（本 README ＋ 編號 00–19 共 20 份）的總覽、
  閱讀路徑、操作守則與目前狀態。
- **本文件不做的事**：不定義任何遊戲機制、型別、公式、字串；凡涉及規則一律「參見 `plan/xx-yyy.md` §n」。

## 3. 設計細節

### 3.1 實作者操作指南（總覽）

開工到出貨的整體流程如下；逐步驟版本見 §5.1，開工序列的可勾選清單見 §7。

1. **開工前**：完整閱讀 00 與本 README。
2. **補齊前置文件**：依 §3.9 的順序撰寫尚缺的 `18-roadmap.md`（開工前必須）與其餘三份（各有定稿期限）。
3. **逐里程碑實作**：每個里程碑開始前，閱讀 `plan/18-roadmap.md` 為該里程碑指定的規格文件，
   然後依規格實作、依 17 的品質門檻驗收。
4. **全程遵守**：§3.2 優先序、§3.3 BAL 規則、§3.4 矛盾處理流程、§3.5 鐵律。
5. **不做設計決策**：規格已把決策做完。若規格真的有缺口或矛盾，走 §3.4 流程，不得停下來「等人決定」。

### 3.2 規格衝突優先序（複述 00 卷首，canonical）

> 所有 `plan/*.md` 文件與未來程式碼，凡與 00 衝突者，以 00 為準。規格衝突的優先序：
> **`00-foundations` > `02-data-model` > `15-balance` > 各系統文件（03~10、16）> UI 文件（11~13）**。

補充裁決細則（皆源自 00）：

- 同層級的系統文件之間發生重疊時，以 00 §7 分工表指定的「單一真相來源」文件為準；
  其他文件的複述若不一致，視為該文件的錯誤，依 §3.4 流程修正該文件。
- 型別與欄位命名一律以 02 為準；02 又承襲 00 §4／§8／§14 的命名。
- BAL 常數值：15 定案值 > 各文件建議初值（見 §3.3）。

### 3.3 BAL 常數規則（複述 00 §11）

- 所有可調數值（係數、成本、門檻、機率、時長）**必須**命名為 `BAL.camelCaseName`，
  集中實作於單一檔案 `src/core/balance.ts`。程式其他地方禁止出現魔術數字。
- 各系統文件在公式中引用常數名並附「建議初值」；`plan/15-balance.md` 彙整成主表。
  若各文件建議值衝突，以 15 的定案值為準。
- `plan/15-balance.md` 已存在；`BAL.*` 名稱與定案值以 15 主表及 `src/core/balance.ts` 對齊結果為準。
  D2 所載過渡規則只保留為文件尚未完成時的歷史裁決，不再用於目前 main。
- 難易度（初級／中級／上級）修正只能作用於 AI 勢力的收入／士氣／AI 積極度乘數，不改玩家規則。

### 3.4 發現規格矛盾或不可行時的處理流程（複述 00 §0 第 5 條）

> 以優先序高的文件為準**做出決定**、**完成實作**，並**回頭修改**對應的 plan 文件、
> 在該文件的「§8 設計決策記錄」註記變更原因。**不得留下未解決的 TBD。**

要點：

- 流程是「決定 → 實作 → 回寫 plan §8」，三步缺一不可；逐步驟版本見 §5.3。
- 修改的對象是**優先序較低、內容有誤的那份文件**；00 本身永不修改內容
  （若 00 真的有誤，僅允許在其他文件 §8 記錄歧異與採行方案，實務上以 00 為準）。
- 回寫時同步更新該文件正文，使正文與 §8 記錄一致；不允許只改 §8 不改正文。

### 3.5 給實作 AI 的十條鐵律（皆為複述，附出處）

1. `src/core/` 是純 TypeScript 遊戲邏輯，**不得 import 任何 React／Pixi／DOM API**；
   core 是確定性狀態機 `(GameState, Command[]) --tick--> GameState + GameEvent[]`（00 §3）。
2. 決定論：core 內禁用 `Date.now`／`Math.random`；全部隨機數走 `rng.ts` 的五個具名流
   `rng.battle / rng.dev / rng.ai / rng.event / rng.misc`（00 §5.5）。
3. 每日 tick 的 13 步系統順序固定，不得增刪換序（00 §5.4；細節 03 §3.2）。
4. 一個機制只在一份文件定義；程式註解與文件引用一律「參見 `plan/xx.md` §n」，不複述公式（00 §7）。
5. 全部可調數值以 `BAL.*` 命名、集中於 `src/core/balance.ts`（00 §11；本文件 §3.3）。
6. 全部 UI 文字經 `t(key)` 取得、唯一語系 `zh-TW`、key 依 `區域.畫面或元件.語意`；
   **全案禁止簡體字**，CI 以黑名單掃描（00 §9；17 §5.4）。
7. ID 一律小寫羅馬字 slug 加型別前綴（`clan.oda`、`off.oda-nobunaga`、`castle.kiyosu`…）；
   專有名詞顯示名放資料 JSON 的 `name` 欄位，不進 i18n 表（00 §8）。
8. 規格衝突優先序 `00 > 02 > 15 > 系統文件 > UI 文件`（00 卷首；本文件 §3.2）。
9. 矛盾處理：決定 → 實作 → 回寫 plan §8；不得留 TBD（00 §0；本文件 §3.4）。
10. 技術選型不得更改、禁止引入 Redux／MobX／Three.js／CSS 框架／任何後端（00 §2）。

### 3.6 回寫 plan 文件時的撰寫規範摘要

依 §3.4 流程回寫任何 plan 文件（含更新 13／15／18／19）時，遵守 00 §13：

- 章節結構固定八段：1 目的與範圍、2 與其他文件的關係、3 設計細節、4 資料結構、
  5 演算法與公式、6 UI/UX（如適用）、7 實作任務清單、8 設計決策記錄。
- 行文：繁體中文（台灣慣用語）、專業直述、無「待定／TBD」；
  程式識別符／型別／常數用英文；表格枚舉；代碼塊用 TypeScript。
- 資料結構：TypeScript interface，每個欄位附註解與單位；命名以 02 為準。
- 公式：數值一律 `BAL.camelCaseName` ＋建議初值；演算法用步驟化虛擬碼。
- §8 記錄格式沿用各文件慣例：一條一決策，含日期、矛盾或取捨內容、決定、理由。
- 術語用字依 00 §14 與 19；提交前通過簡體字掃描（17 §5.4）。
- **絕對不要修改 `00-foundations.md`**（§3.4、§5.3 第 5 步）。

### 3.7 常見實作陷阱（前置提醒，內容皆出自各文件）

1. **決定論破口**：在 core 內偷用 `Math.random`、`Date.now`、`Object.keys` 未排序迭代、
   `Set`／`Map` 插入序依賴——golden test（17 §3.5）會抓到，但修起來昂貴；一開始就避免。
   迭代順序與 canonical stringify 規範見 02 §3.4、§5.4。
2. **跨文件複述走樣**：實作時發現兩份文件對同一機制描述不一致，先查 00 §7 誰是擁有者，
   再走 §5.3；不要「取平均」或自行折衷出第三種行為。
3. **月結順序**：每月 1 日多個系統都有動作（收入、忠誠、具申、AI 評定……），
   順序錯誤會導致數值軌跡與 golden 快照不合；以 00 §5.4 的 13 步與 03 §3.6 月結展開為準。
4. **魔術數字滲漏**：UI 層的動畫時長、渲染閾值不屬於 BAL（歸 12 tokens 與 04 §4.5
   的渲染設定常數）；但凡影響遊戲模擬結果的數字都必須進 `balance.ts`。分界判準：
   「改了這個數，golden hash 會不會變」——會變者屬 BAL。
5. **字串硬編碼**：任何繁中文字出現在 `.tsx` 字面量即違規（00 §9）；
   專有名詞（人名城名郡名）例外——它們在劇本 JSON 的 `name` 欄位（00 §8）。

### 5.2 BAL 常數取值查找順序

```
resolveBalValue(constantName):
  1. 若 plan/15-balance.md 已定稿且主表含此常數 → 用 15 的定案值。
  2. 否則查 00 §7 分工表，找出擁有該機制的文件 D → 用 D 的 §5 建議初值。
  3. 若 D 未給值、但其他文件給了 → 用該文件建議值，並依 §5.3 回寫 D 補上。
  4. 若全部文件皆未給值 → 這是規格缺口：自行取一個符合 00 §6 平衡設計目標的值，
     依 §5.3 記錄於擁有文件的 §8，並於 15 撰寫時彙入主表。
  （任何情況下：值只寫進 src/core/balance.ts 一處。）
```

### 5.3 規格矛盾處理流程（00 §0 第 5 條的逐步化）

```
onSpecConflict(docA, docB, issue):
  1. 判定權威：依 §3.2 優先序；同層級則依 00 §7 分工表找機制擁有者。
  2. 做出決定：採權威文件的版本；若權威文件本身不可行，
     設計一個與 00 的 canonical 原則相容的最小修正方案。
  3. 完成實作：依決定寫程式與測試，不留 TBD、不留註解式問號。
  4. 回寫文件：修改內容有誤（優先序較低）的那份文件的正文，
     並在同一份文件的「§8 設計決策記錄」新增一條：日期、矛盾內容、決定、理由。
  5. 00-foundations.md 永不修改；涉及 00 的歧異僅記錄於他文件 §8 並以 00 為準。
```

### 5.4 UI 字串新增流程

```
addUiString(key, text):
  1. key 依 00 §9 規範（區域.畫面或元件.語意；插值用 {name}）。
  2. 優先沿用來源文件 §6 字串表既有的 key 與文案，不自創同義字串。
  3. 寫入 src/i18n/zh-TW.ts 單一檔案；元件中一律 t(key)。
  4. 文案為繁體中文（台灣慣用語）；數字格式依 00 §9（如 3,000兵、12,500石、800貫）。
  5. 通過簡體字掃描（17 §5.4）；13 定稿後以 13 主表為準對齊。
```
